# Signal Track 需求说明

## 产品目标

Signal Track 是一个在 Windows Codex App 内运行的投资信号跟踪工作流。用户把信息源内容通过对话或文件拖拽交给 Codex，Codex 负责理解内容、抽取标的和操作、判断跟踪逻辑是否足够，并调用本仓库的本地工具完成落库、检查、渲染和发布。

本项目不提供后台分析服务，也不在后台服务中调用大模型。所有 AI 分析都由当前 Codex 对话完成。

## 输入方式

- 用户在 Codex App 对话里粘贴信息，或拖拽文本、Markdown、Word、PDF 等文件。
- 如果输入中没有明确的信息源，Codex 必须先向用户确认信息源，再分析内容。
- 一条输入可以包含一个或多个标的。
- 如果用户明确说明多个标的是一个组合，则合并为一个组合跟踪项目。
- 如果组合没有给出各标的权重，Codex 必须确认权重，因为权重决定组合曲线和收益统计。

## 信号抽取

Codex 需要从输入中识别：

- 信息源名称。
- 标的名称、代码、交易市场。
- 操作：开仓、平仓、更新、仅观察、无操作。
- 方向：做多、做空、中性或未知。
- 原始开仓逻辑。
- 后续观察逻辑。
- 平仓或复核触发条件。
- 是否为组合，以及组合权重。

如果一条信息包含多个独立标的，默认生成多个跟踪项目。如果用户明确说明是组合，生成一个组合项目。

## 跟踪逻辑补充

Codex 需要自动判断原始跟踪逻辑是否足够执行每日检查。

- 逻辑足够时，优先使用信息源里的原始逻辑。
- 逻辑不足或过于简单时，不再询问用户是否跟踪，直接跟踪。
- 逻辑不足时，Codex 使用用户给出的 3C-5M-3D-3T 投研框架补充观察指标、风险条件和退出触发条件。
- 补充逻辑必须清楚标注为系统补充逻辑，避免和信息源原文混淆。
- 涉及财务、行业、估值或新闻数据时，应优先联网核验并保留来源记录。

## 每日检查

每日检查由 Codex App Automations 定时触发，不由后台服务或 systemd 触发。

建议调度：

- 19:00 Asia/Shanghai：主检查，覆盖 A 股、港股、中国期货和一般项目。
- 07:00 Asia/Shanghai：可选补跑，覆盖美股和美股期货。

检查流程：

1. 刷新必要行情数据。
2. 检查所有 active 跟踪项目。
3. 根据观察逻辑和价格表现生成每日检查记录。
4. 若触发平仓或强复核条件，生成平仓信号。
5. 重新生成静态 HTML dashboard。
6. 调用外部演示发布 API 覆盖发布网页。

## 行情能力

系统应支持以下市场的历史价格获取：

- A 股。
- 港股。
- 中国市场期货。
- 美股。
- 港股相关期货。
- 美股期货。

正式行情源可由 Tushare、yfinance 或后续 licensed provider 提供。当前架构通过 provider 接口隔离数据源。

## 网页展示

发布物必须是纯前端静态 HTML。网页只负责展示，不承担分析、调度或后端 API 逻辑。

网页需要支持：

- 电脑端横屏排版。
- 手机端竖屏排版。
- 按信息源分组和筛选。
- 全部项目列表。
- 活跃项目、平仓信号、已关闭项目视图。
- 单标的价格曲线：从开仓前一个月，到平仓后一个月或当前日期。
- 组合项目曲线：组合加权收益曲线，以及每个 leg 的单独曲线。
- 每个项目的原始逻辑、系统补充逻辑、每日检查历史和最新动作。

视觉方向遵循 `DESIGN.md`：Card based design with layered elements，Futuristic minimalism with glassmorphism。

## 数据架构

系统采用 SQLite + Markdown 的混合架构。

- SQLite 是运行状态的事实源，保存项目、标的、组合权重、逻辑块、每日检查、平仓信号、价格数据和发布记录。
- Markdown 是人类可读的分析档案，保存投研报告、3C-5M-3D-3T 补充分析、重要原文和复盘。
- 网页从 SQLite 的结构化状态和 Markdown 摘要渲染，不依赖重新解析 Markdown 来判断项目状态。
- Codex 生成的长文分析可以写成 Markdown，但关键触发条件、观察指标和项目状态必须同步进入 SQLite。

详细设计见 `docs/数据架构.md`。

## 明确不做

- 不启动 FastAPI 或其他后台服务。
- 不提供 Web Inbox。
- 不在后台服务中调用 OpenAI 或其他大模型 API。
- 不使用 systemd 或本机常驻 scheduler 做每日检查。
- 不让发布后的网页承担分析工作。
