# Signal Track 开发计划

## 当前产品形态

Signal Track 采用 Codex App-first 架构：

1. 用户在 Windows Codex App 中通过对话或拖拽文件输入信息。
2. Codex 使用项目本地 skill 完成 AI 分析、结构化抽取和逻辑补充。
3. Codex 调用本地 CLI 更新 SQLite、刷新行情、生成静态 HTML。
4. Codex 调用发布 API 覆盖发布纯前端 dashboard。
5. 每日检查通过 Codex App Automations 定时触发。

这里的 Codex App-first 指：Codex App 是唯一的 AI 分析、调度和执行入口。本项目不再保留后台 FastAPI 服务、后台 OpenAI 调用、Web Inbox 或 systemd 定时器。

## 保留模块

- `db.py`：SQLite schema、迁移、读写。
- `signals.py`：项目创建、组合、权重、开平仓匹配。
- `checker.py` / `rules.py`：每日检查、收益和规则触发。
- `providers/`：fixture、Tushare、yfinance、auto provider。
- `dashboard.py`：静态 HTML dashboard。
- `publisher.py`：演示管理发布 API 客户端。
- `cli.py`：Codex 调用的本地工具入口。
- `.codex/skills/signal-track/`：项目本地 Codex skill。

## 明确边界

- 本仓库不启动常驻后台服务。
- 发布 API 只是外部静态 HTML 上传目标，不是 Signal Track 的分析后台。
- 发布后的网页只展示数据，不做 AI 分析、不调度任务、不写入数据库。
- 每次新输入、每日检查和发布都由 Codex App 对话或 Automations 驱动。

## 数据架构决策

采用 SQLite + Markdown 的混合架构：

- SQLite 作为运行状态的 canonical state，保存项目、标的、组合权重、逻辑块、每日检查、平仓信号、价格数据和发布事件。
- Markdown 作为人类可读的投研档案，保存长文报告、系统补充逻辑、阶段复盘和重要原文。
- dashboard 从 SQLite 读取结构化状态，并可嵌入或链接 Markdown 摘要。
- 不采用 Markdown-only 架构，避免每日检查时靠重新解析文本推断项目状态。

后续开发需要补强报告索引：在 SQLite 中记录 Markdown 报告路径、版本、hash 和关联项目。

## 行情范围

- A 股：Tushare。
- 港股：Tushare，yfinance fallback。
- 中国期货：Tushare。
- 美股：Tushare 或 yfinance。
- 港股期货/美股期货：yfinance 作为临时 fallback；正式环境应接 licensed provider。

## 每日检查

建议用 Codex App 自动化配置：

- 19:00 Asia/Shanghai：主检查和发布。
- 07:00 Asia/Shanghai：可选 US 市场补跑。

自动化运行：

```powershell
python -m signal_track.cli daily-run --provider auto --publish
```

如果行情 provider 不可用，降级：

```powershell
python -m signal_track.cli daily-run --provider none --publish
```

## 网页要求

- 发布物是单个静态 HTML。
- 支持电脑端横屏和手机端竖屏。
- 手机端不能页面级横向溢出；表格可在容器内横向滚动。
- 组合项目显示组合收益曲线和每个 leg 的单独曲线。
- 视觉遵循 `DESIGN.md`：Card based design with layered elements，
  Futuristic minimalism with glassmorphism。
