# Signal Track 数据架构

## 核心结论

Signal Track 采用 SQLite + Markdown 的混合数据架构：

- SQLite 是运行状态的事实源。
- Markdown 是人类可读的投研分析、复盘和审计档案。

不要使用 Markdown 作为唯一数据源来反复解析系统状态。Codex 可以阅读和生成 Markdown，但项目状态、组合权重、每日检查、价格曲线和发布记录必须有结构化存储。

## 为什么不是纯 Markdown

本项目的大量内容是文字分析，但它不是一个纯文档库。它有明确的状态流转和时间序列：

- 开仓、更新、平仓。
- 多信息源、多标的、多组合。
- 每日检查和退出信号。
- 组合权重和加权收益曲线。
- 历史价格数据。
- 网页发布记录。

这些数据需要稳定查询、过滤、聚合、计算和渲染。若全部放在 Markdown 中，系统每天都需要重新读取大量文本并推断状态，容易出现歧义和回归。

## SQLite 保存什么

SQLite 保存机器需要稳定处理的 canonical state：

- 原始输入记录：信息源、时间、原文、文件路径或文件 hash。
- 标的主数据：代码、市场、名称、别名。
- 跟踪项目：状态、方向、开仓日期、平仓日期、信息源。
- 组合 leg：标的、方向、权重。
- 逻辑块：原始逻辑、系统补充逻辑、更新逻辑、手动备注。
- 每日检查记录：检查日期、观察结果、触发条件、决策。
- 平仓信号：触发时间、原因、状态。
- 历史价格：OHLCV、数据源、日期。
- 发布事件：发布 URL、状态码、错误信息、时间。
- Markdown 报告索引：`project_reports` 记录报告路径、格式、hash、大小和关联项目。

## Markdown 保存什么

Markdown 保存人要阅读、复盘和审计的长文本：

- 每个项目的投研报告。
- 3C-5M-3D-3T 补充分析。
- 重要信息源原文归档。
- 阶段性复盘。
- 平仓复盘。
- 需求、设计、开发计划和运行手册。

Markdown 可以被网页展示，也可以被 Codex 用作上下文，但不应该作为状态判断的唯一来源。

## 推荐数据流

1. 用户把信息交给 Codex。
2. Codex 分析并生成结构化抽取结果。
3. CLI 将结构化结果写入 SQLite。
4. 必要时生成或更新 Markdown 投研档案。
5. 每日检查从 SQLite 读取项目、逻辑和价格数据。
6. 检查结果写回 SQLite。
7. dashboard 从 SQLite 结构化数据和 Markdown 摘要渲染静态 HTML。
8. Codex 调用外部发布 API 上传静态 HTML。

## 设计原则

- 系统状态以 SQLite 为准。
- 长文本分析以 Markdown 保存，便于阅读和 git diff。
- 网页渲染不要重新从 Markdown 猜项目状态。
- Codex 可以补充分析，但补充后的关键指标和触发条件要结构化入库。
- 数据库中应保留 Markdown 文件路径和 hash，保证报告和项目状态可追溯。
